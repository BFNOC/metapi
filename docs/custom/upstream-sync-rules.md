# 上游 PR 跟随规则

> 适用范围：本 fork 后续继续跟随上游 `cita-777/metapi` 的 PR / commit 时，统一采用本规则做分诊、实施、验证与记录。
>
> 当前规则基于近期已完成的 `#422/#439`、`#426`、`#429` 等 repo-local 手工移植实践总结而来。

---

## 1. 目标

这份规则要解决的不是“怎么把上游 patch 搬过来”，而是：

1. 如何快速判断一个上游 PR 值不值得跟
2. 如何避免把上游的大重构、owner 迁移、UI 分叉机械带进本 fork
3. 如何在保留 fork 现有语义的前提下，吸收真正有价值的边界修复和能力补齐
4. 如何把实施结果沉淀回 `docs/custom/upstream-sync-log.md`，形成可复用的跟随链路

---

## 2. 核心原则

### 2.1 docs-first，不先冲 GitHub

做 upstream PR 跟随时，默认先读本地资料：

- `docs/custom/README.md`
- `docs/custom/upstream-sync-log.md`
- `docs/reviews/*`
- 用户点名提供的本地计划文档

只有在本地文档无法回答这些问题时，才继续抓上游 patch / PR：

- 这个 PR 是不是已经评过
- 是否已有本地等价覆盖
- 用户之前是否已明确说过“继续跳过”
- 当前 fork 的 owner 和语义是否已经和上游分叉

### 2.2 owner-first，不机械照抄 patch

真正实施时，优先找本 fork 当前 owner：

- 哪个文件真正拥有这段语义
- 哪个测试文件最接近回归面
- 哪些文件是高冲突热区，不适合整包 cherry-pick

结论先按 owner 写，再决定是否吸收 upstream patch 中的具体实现。

### 2.3 selective port，不做“整包跟随”

默认假设：

- 上游 patch 里的“有效部分”和值得合入的范围，通常小于整个 PR

优先吸收：

- 边界修复
- bug fix
- 当前 fork 已有能力上的直接缺口
- 与当前 owner 明确对齐的小范围测试

默认不顺手吸收：

- owner 迁移
- 目录重排
- 新体系引入
- 大规模 UI 重构
- 与当前部署场景无关的管理面扩展

### 2.4 本地语义优先

当上游行为和本地语义冲突时：

- 优先保留本地已成型的语义
- 不为“和上游看起来一致”而回退本 fork 已验证有效的行为

例外：

- 用户明确要求“完全跟随上游”
- 或当前问题本身就是因为本地偏离上游而引入的回归

### 2.5 结论必须可验证

每次跟随都要回答这几个问题：

1. 为什么跟 / 不跟
2. 是 cherry-pick、手工移植，还是继续跳过
3. 真正改了哪些 owner
4. 用哪些测试证明这次修改有效
5. 有没有把基线噪音误记成当前回归

## 3. 标准流程

### 3.1 第一步：先做本地分诊

默认流程：

1. 读 `docs/custom/upstream-sync-log.md`
2. 看该 PR / commit 是否已在：
   - 已完成
   - 已覆盖
   - 第二批候选
   - 明确跳过
3. 对照当前代码 reality，确认文档结论是否仍成立

常见输出分类：

- 已有本地等价覆盖
- 建议优先合入
- 第二批候选
- 继续跳过

### 3.2 第二步：再抓上游 patch

只有分诊后仍值得继续时，才去抓：

- PR patch
- 相关 follow-up commit
- upstream `main` 上同文件的最新版本

抓 patch 时，不只看“新增了什么”，还要拆成：

- 真正有价值的功能点
- review 后追加的结构整理
- 与本 fork 无关的周边调整

同时检查这条 PR 后面有没有直接 follow-up：

- 同作者或 review 后补上的修正 commit
- 同一问题在后续 PR 里的补丁
- upstream `main` 当前同文件是否已经继续收口边界

不要把“上游中间态”机械搬进本 fork。

### 3.3 第三步：决定合入方式

每个 PR 只在这三种方式里选一种：

### A. 直接 cherry-pick

只在同时满足以下条件时考虑：

- 改动面小
- 本地对应文件分叉不大
- 没有 schema / runtime settings / UI / route owner 联动
- 不会把无关体系顺手带进来

### B. repo-local 手工移植

这是默认首选，适用于：

- 本地 owner 已明显分叉
- 上游 patch 价值集中在局部能力点
- 需要保留本地语义
- 需要拆成 phase 落地

### C. 明确跳过

跳过不是“不看”，而是看完后明确记录原因，例如：

- 与当前部署场景不匹配
- 依赖此前已跳过的大体系
- 本地已有等价覆盖
- 主要是 UI / 桌面 / 文案调整
- 当前收益不够抵消冲突面

## 4. 何时写计划文档

满足以下任一条件时，先写 plan 再开工：

- 改动跨多个 owner
- 有“跟上游”与“保留本地语义”的决策门
- 用户明确要求先写 `docs/custom/plans/*`
- 有外部 reviewer / Claude / Gemini 的方案需要 repo-local 校正

计划文档至少要写清楚：

1. 当前 fork 现状
2. 上游 patch 实际包含什么
3. 本次只跟哪部分，不跟哪部分
4. phase 顺序
5. 风险与防呆
6. 测试与验证计划

注意：

- `docs/custom/plans/` 现在是 **ignored 的本地计划目录**
- 它可作为输入材料，但不要把它误当成正式 tracked 文档体系

## 5. 实施规则

### 5.1 只改真正 owning 的位置

优先改：

- 当前 fork 真正拥有语义的模块
- 紧邻该 owner 的测试文件

不要为了“跟上游文件布局一致”而顺手：

- 移动模块
- 大范围调 import
- 下沉 / 上提 owner

除非这次任务的目标本身就是结构迁移。

### 5.2 先做最小闭环

推荐顺序：

1. 先补核心实现
2. 立刻补对应测试
3. 只在第一层闭环稳定后，再补设置项 / UI / 后续扩展

如果 PR 可拆 phase，默认拆成：

- P0：核心行为修复
- P1：运行时开关 / 设置 / UI / 运营补完

### 5.3 大任务默认 subagents，并由主线程总集成

适用场景：

- 多文件、可拆 owner、互不重叠的实施任务

执行方式：

1. 主线程先拆 owner 和边界
2. worker 只拿自己负责的文件
3. 主线程负责：
   - 看 `git status` / `git diff`
   - 清理重复改动
   - 修 test harness 漂移
   - 跑最终验证
   - 做最后审批

若 worker 卡住：

- 不空等
- 主线程应尽快接管关键路径

## 6. 测试与验证规则

### 6.1 默认验证口径

每次 upstream 跟随，至少做：

- 紧邻 owner 的 targeted tests
- 新增 / 修改测试对应的 suite
- `npm run repo:drift-check`

按改动范围补充：

- 若改到共享运行时边界，再补 typecheck
- 若改到 settings / 前端，再补对应 web suite
- 若改到 schema / migration，再补迁移或兼容性测试

### 6.2 区分三类失败

验证失败后，先分清：

### A. 当前补丁回归

- 只有本次改动后才出现
- 与改动 owner 直接相关

### B. 基线噪音

- 合入前就存在
- 或与本次功能无直接关系

### C. test harness 漂移

- 业务代码是对的，但测试夹具 / mock 没跟上当前依赖链

这种情况允许做最小测试桩修复，但需要满足：

- 只修测试文件
- 不改生产语义
- 尽量参考 upstream `main` 当前同类测试的最新 harness

### 6.3 验证结果要写进记录

sync-log 中不要只写“已验证”，而要写清楚：

- 跑了哪些命令
- 各 suite 通过多少条
- `drift-check` 是否有 violation
- 是否有基线债未解决

## 7. 文档更新规则

### 7.1 正式结论优先写入 `upstream-sync-log.md`

当一个 PR 完成以下任一动作后，应优先更新：

- 已决定跳过
- 已决定跟随方式
- 已完成 repo-local 手工移植
- 已确认本地等价覆盖

统一落点：

- `docs/custom/upstream-sync-log.md`

### 7.2 记录方式

优先 append-only：

- 旧的审阅表格不重写
- 在文末新增“实施记录 / 补充说明”
- 候选 / 待定 / 跳过的旧判断保留历史，不回写成“仿佛一开始就已完成”

推荐记录结构：

1. 处理结果
2. 本次落地的上游 commit
3. 本地实现说明
4. repo-local 差异点
5. 验证结果
6. 结论

### 7.3 计划文档与正式记录分层

- `docs/custom/plans/*`：本地计划输入
- `docs/custom/upstream-sync-log.md`：正式同步记录

不要把“计划里说完成了”直接等同于“sync-log 已完成”。

## 8. 提交规则

近期已经验证有效的提交流程：

### 8.1 代码和文档拆开提交

推荐拆成两次：

1. 代码提交
2. `sync-log` 文档提交

优点：

- review 更清楚
- 回滚更干净
- 与近期 `#426`、`#429` 的历史风格一致

### 8.2 提交信息风格

优先中文，多行 body，总结：

- 改了哪些 owner
- 为什么这样改
- 验证跑了什么

常用前缀：

- `fix:` bug / 边界修复 / selective port
- `feat:` 新能力
- `docs:` sync-log 或说明文档更新
- `refactor:` 纯结构整理（不和功能修复混提）

## 9. 近期可复用的判断模板

### 9.1 值得跟的信号

- 当前代码存在直接缺口
- patch 价值集中在边界修复
- 改动 owner 清晰
- 可用 targeted tests 快速兜住

### 9.2 暂缓的信号

- 价值存在，但需要真实流量 / 痛点来证明
- patch 同时带大量设置项、UI、运营逻辑
- 现阶段不是最紧急收益点

### 9.3 跳过的信号

- 依赖此前已跳过的大体系
- 本地已有明显分叉且收益低
- 主要是 UI / 桌面 / release 工程调整
- 与当前 fork 的部署场景不匹配

---

## 10. 后续跟随的执行清单

每次继续跟新的上游 PR 前，按这个清单走：

1. 先读 `docs/custom/upstream-sync-log.md`
2. 再看当前 fork owner 和测试 reality
3. 再抓上游 patch / follow-up
4. 先判断是 cherry-pick、手工移植，还是跳过
5. 大任务先写 plan
6. 实施时只改真正 owner
7. 跑 targeted tests + `repo:drift-check`
8. 先更新 `sync-log`
9. 再按“代码 / 文档”拆 commit

---

## 11. 当前结论

对本 fork 来说，后续跟上游 PR 的默认策略已经可以固定为：

- **先本地文档，后上游 patch**
- **先 owner reality，后 implementation**
- **默认 selective port，不默认 cherry-pick**
- **本地语义优先，但可按用户要求完全跟随上游**
- **验证结果必须写回 `upstream-sync-log.md`**
- **大任务由主线程总监督，subagents 并行实施**
