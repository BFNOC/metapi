# 路由健康模型

本文档描述 Metapi 路由引擎的站点健康评估策略。

## 设计原则

路由健康度**纯粹由错误信号驱动**。延迟指标仅用于 WebUI 展示和人工观测，不参与路由权重计算。

### 为什么不用延迟？

Metapi 的典型使用场景是聚合 30+ 个公益/中转代理站点。这些站点的延迟特征使得延迟惩罚容易产生误判：

- **思考型模型**（o1、Claude Reasoning）首字延迟可达 30-120 秒，是模型特性而非站点慢
- **非流式请求**（embeddings、images）总延迟取决于 payload 大小
- **模型混用**：同一个站点同时服务 gpt-4o-mini（快）和 o1-pro（慢），全局延迟指标必然误导
- **中转站延迟透传**：公益站大多是 New-API/One-API 风格的纯转发，延迟主要由上游决定

因此，本系统选择了更稳健的策略：**只要不出错就算健康。**

## 健康评估架构

### 两层评估

#### 1. 运行时健康（实时、内存中）

每个 `(站点, 模型)` 组合维护一个独立的 `SiteRuntimeHealthState`（单桶模型，无全局桶）：

| 字段 | 含义 |
|------|------|
| `penaltyScore` | 累积的错误惩罚分数，半衰期 10 分钟自动衰减 |
| `recentSuccessStreak` | 最近连续成功次数，用于抑制软错误惩罚 |
| `breakerLevel` | 熔断等级（0-3），连续瞬态错误触发升级 |
| `breakerUntilMs` | 熔断解除时间戳 |
| `latencyEmaMs` | 延迟 EMA（**仅观测**） |

**运行时权重乘数计算**（纯错误驱动）：

```
if 熔断中 → multiplier = 0.02（近零）
else → multiplier = 1 / (1 + penaltyScore)
```

不引用 `latencyEmaMs`。

#### 2. 历史健康（持久化，基于 DB 统计）

从路由通道表（`route_channels`）聚合每个站点下所有通道的 `successCount / failCount`：

```
successRate = successCount / totalCalls
multiplier = 1 - ((1 - successRate) × 0.55 × sampleFactor)
```

同样不引用历史平均延迟。

### 最终权重计算

```
finalWeight = baseContribution / sameChannelCount  // 同站点通道均分
  × combinedSiteWeight        // 站点配置权重 × 下游策略倍率
  × runtimeHealthMultiplier   // 运行时错误惩罚乘数
  × historicalMultiplier      // 历史成功率乘数
```

## 错误信号分类

错误按**健康维度**分为两大类：

### 接入健康（Admission Health）—— 站点能否成功接入

这些错误表示站点当前不可用或拒绝服务，应重罚并可触发熔断。

| 错误类型 | Penalty | 触发熔断 | 说明 |
|----------|---------|---------|------|
| 401/403（凭证失效/封禁） | 4.0 | 是 | 站点配置问题或 IP 封禁 |
| 402（额度耗尽） | 5.0 | 是 | 号池见底 |
| 429（限流） | 2.2 | 是 | 并发打满，短冷却 |
| 5xx（服务器错误） | 2.5 | 是 | 网关/上游不可达 |
| 连接失败/超时 | 2.5 | 是 | 网络层问题 |
| quota exhausted / 无可用渠道 | 2.5 | 是 | 识别为瞬态错误 |
| 协议不支持 | 0.6 | 否 | 端点路径不匹配 |
| 模型不支持 | 0.9 | 否 | 模型未上架 |
| 请求验证失败 | 0.25 | 否 | 参数格式问题 |
| 其他 4xx | 0.9 | 否 | 未分类客户端错误 |

### 完成健康（Completion Health）—— 接入后能否稳定完成

这些错误表示站点是活的、HTTP 也成功了，但内容传输过程中出问题。
**不应重罚、不触发熔断。**

| 错误类型 | Penalty | 触发熔断 | 说明 |
|----------|---------|---------|------|
| stream closed before response.completed | 0.30 | 否 | 流中途断开，上游账号池耗尽或网络抖动 |
| upstream stream failed | 0.30 | 否 | 上游流处理失败 |
| upstream returned empty content | 0.35 | 否 | 站点响应但无有效输出 |
| 客户端取消（AbortError） | **0** | 否 | 下游主动断开，不是站点的错 |
| 未分类默认 | 1.2 | 否 | 兜底 |

> **设计原理**：一个 96% 成功率的站点不应因 2 次 "stream closed" 就被打到 ×0.30。
> 改进前：penalty=1.2 × 2 → multiplier = 0.29（误杀）
> 改进后（分类+抑制）：见下方成功信任机制

### 成功信任抑制机制

对于**完成健康类**软错误，惩罚会被站点的**最近成功连续数（recentSuccessStreak）**抑制：

```
effectivePenalty = basePenalty × 1/(1 + recentSuccessStreak / K)
```

其中 K=10（可配置）。效果：

| 最近成功次数 | 抑制系数 | stream closed 有效惩罚 | 3次后 multiplier |
|-------------|---------|----------------------|-----------------|
| 0（刚启动） | 1.00 | 0.30 | 0.53 |
| 10 | 0.50 | 0.15 | 0.69 |
| 20 | 0.33 | 0.10 | 0.77 |
| 50 | 0.17 | 0.05 | 0.87 |
| 100 | 0.09 | 0.03 | 0.92 |

关键规则：
- **软错误不重置 streak** — 站点本质上是健康的，信任不应被偶发中断清零
- **硬错误立即重置 streak** — 403/5xx 等接入失败说明站点状态变了
- 客户端取消（AbortError）完全跳过，不影响任何状态

### 单桶模型（按站点+模型）

每个 `(siteId, modelName)` 维护一个独立的健康桶。没有站点全局桶。

- 错误只记录到对应模型的桶
- 成功只恢复对应模型的桶
- 不同模型之间互不影响

`multiplier` 直接就是该桶的 `1 / (1 + penaltyScore)`，无乘法叠加。

### 通道级冷却与错误分类的协同

通道级 `recordFailure` 根据错误类型采取不同策略：

| 错误类型 | failCount | consecutiveFailCount | cooldownUntil | 说明 |
|----------|-----------|---------------------|---------------|------|
| 客户端取消 | 不变 | 不变 | 不变 | 完全跳过通道级记录 |
| 软完成失败 | +1 | 不变 | 10 秒固定 | 短暂避让，不触发 fibonacci 升级 |
| 硬失败 | +1 | +1 | fibonacci 退避 | 完整冷却，60s × fib(n) |

### 手动重置

管理员可通过两种粒度的 API 清除冷却和惩罚状态：

#### 站点级重置

`POST /api/sites/:siteId/reset-health`

- 删除该站点下所有模型桶的内存状态（penalty、breaker）
- **同步清除**该站点下所有通道的 DB 级冷却字段（`cooldownUntil`、`cooldownLevel`、`consecutiveFailCount`、`lastFailAt`）
- 更新内存通道缓存（`patchCachedChannel`）
- **立即持久化**（不走 debounce），确保进程重启后不会恢复旧惩罚
- 清除受影响路由的决策快照（包括 explicit_group 依赖路由）
- 前端自动刷新决策展示

#### 通道级冷却重置

`POST /api/channels/:channelId/reset-cooldown`

- 仅清除指定通道的 DB 级冷却字段，不影响同站点其他通道
- 适用场景：单个通道因偶发错误进入冷却，但站点整体健康
- 清除受影响路由的决策快照
- 刷新路由缓存

> 对于黑与白这样 573 成功的主力站点，即使连续 5 次 stream closed，
> 有效惩罚仅 5 × 0.005 = 0.025，multiplier 仍保持 ×0.98。

### 熔断机制

连续 3 次**接入健康类**瞬态错误触发熔断升级（完成健康类错误不计入）：

| 等级 | 冷却时间 |
|------|----------|
| 0 | 无 |
| 1 | 3 分钟 |
| 2 | 15 分钟 |
| 3 | 60 分钟 |

### 恢复机制

成功请求触发渐进恢复：
- `penaltyScore × 0.5`（减半）
- `breakerLevel - 1`（降一级）

## WebUI 可视化

在路由管理页面（`/routes`），每个通道的决策快照中显示运行时健康标签：

| 标签 | 颜色 | 含义 |
|------|------|------|
| 🔴 熔断 | 红色 | 熔断器已触发，暂时不被选中 |
| ⚠ 降权 ×0.xx | 橙色 | 错误惩罚导致权重显著降低（multiplier < 0.5） |
| ⏱ 延迟 Nms | 灰色 | 延迟 EMA 观测值（**仅展示，不参与路由**） |

查看方式：进入路由管理 → 展开某个路由 → 查看各通道右侧的 badge。

### 操作按钮

当通道或站点处于异常状态时，在通道行内会出现可点击的操作按钮：

| 按钮 | 触发条件 | 样式 | 操作 |
|------|---------|------|------|
| ↻ 撤销处罚 | 站点存在运行时惩罚（penalty/breaker） | 橙色渐变药丸 | 调用站点级重置 API |
| ❄ 解除冷却 | 通道处于冷却中状态 | 蓝色渐变药丸 | 调用通道级冷却重置 API |

两个按钮均带有 hover 浮起动效和 tooltip 提示。点击后自动刷新路由决策。

### 算法可观测性

选中概率条（probability bar）和百分比数字上带有 tooltip，hover 后显示路由引擎的完整决策逻辑（如 `weighted-random` 权重分配、`stable-first` 排序依据等），方便诊断通道选择行为。

## 与上游（原版 metapi）的差异

| 特性 | 原版 | 本 Fork |
|------|------|---------|
| 运行时延迟惩罚 | 有（基线 2.5s，最大 35%） | **已移除** |
| 历史延迟惩罚 | 有（基线 2s，最大 18%） | **已移除** |
| 延迟 EMA 记录 | 有 | 保留（仅观测） |
| 熔断窗口 | 1m / 5m / 30m | 3m / 15m / 60m |
| 失败退避基数 | 15s | 60s |
| 错误分层 | 无（统一惩罚） | ✅ 接入健康 vs 完成健康 |
| 软流中断惩罚 | 1.2（默认） | 0.30（轻罚） |
| 客户端取消惩罚 | 1.2（默认） | 0（不罚） |
| 健康 WebUI 可视化 | 无 | ✅ badge + tooltip |
| 手动站点重置 | 仅内存 | ✅ 内存 + DB 通道冷却同步 |
| 手动通道重置 | 无 | ✅ 单通道冷却独立重置 |
| 算法可观测 | 无 | ✅ 概率条 tooltip 显示路由算法 |
| 操作按钮 | 无 | ✅ 撤销处罚 + 解除冷却药丸按钮 |
